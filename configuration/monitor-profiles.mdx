title: "Monitor weights and profiles"
description: "Local DEFAULT_WEIGHTS for the 6-monitor suite, and how server-side task profiles drive monitor steering."
---

ReasonBlocks evaluates trajectory health through six heuristic monitors. The SDK ships local `DEFAULT_WEIGHTS` for `evaluate_all` and for `TokenSavingMiddleware`'s early-exit signals. Separately, `task_profile` (`"coding"` by default, with `"pr_review"` and `"qa"` available) is forwarded to `POST /monitors/evaluate`, where rb-api applies its own per-profile weight tables.

<Note>
  The local `DEFAULT_WEIGHTS` in `reasonblocks.monitors.suite` and the server-side per-profile weights in rb-api are **separate**. The local weights drive `TokenSavingMiddleware`'s early-exit suite and any caller of `evaluate_all`. The server-side weights drive monitor steering injection through `MonitorSteeringInjection`.
</Note>

## The six monitors

Each monitor returns a score in `[0, 1]` where `1` is maximum badness. The weighted sum is the composite. Per-monitor fire threshold (default): `0.6`.

| Monitor | What it detects |
|---|---|
| `streak` | Same action called consecutively (cap=5 returns 1.0) |
| `call_count` | Total tool-call budget consumed (default budget=20) |
| `edit_revert` | Edit-fail-edit thrashing or content-similar reverts on the same file |
| `test_repeat` | Same normalized test-failure signature appearing two or more times in a row |
| `diversity` | Last 5 calls use Ōēż2 distinct tools and the run has Ōēź8 calls (excluding healthy alternation) |
| `hedge` | Hedge-density rising 2x+ between early/late halves, or any explicit retraction phrase |

## Local DEFAULT_WEIGHTS

These are the weights baked into `reasonblocks.monitors.suite.DEFAULT_WEIGHTS`. They drive any direct call to `evaluate_all` and the `signals_fn=default_suite_signals` path used by `TokenSavingMiddleware`.

| Monitor | Weight |
|---|---|
| `streak` | `0.35` |
| `call_count` | `0.15` |
| `edit_revert` | `0.15` |
| `test_repeat` | `0.15` |
| `diversity` | `0.10` |
| `hedge` | `0.10` |

`streak` carries the highest weight because consecutive identical actions are the most reliable stuck-agent signal.

```python
from reasonblocks import DEFAULT_WEIGHTS, DEFAULT_FIRE_THRESHOLD, evaluate_all

steps = [
    {"step_index": 0, "action": "read_file", "thought": "Let me look", "observation": "..."},
    {"step_index": 1, "action": "read_file", "thought": "Hmm, not sure", "observation": "..."},
    {"step_index": 2, "action": "read_file", "thought": "Maybe I missed something", "observation": "..."},
]

result = evaluate_all(steps)
print(result["scores"])    # per-monitor floats
print(result["total"])     # weighted composite
print(result["fired"])     # monitors >= DEFAULT_FIRE_THRESHOLD (0.6)
print(result["weights"])   # weights actually used
```

To override locally, pass a `weights` dict ŌĆö partial dicts merge over `DEFAULT_WEIGHTS`:

```python
result = evaluate_all(steps, weights={"streak": 0.5, "hedge": 0.05})
```

## Server-side task profiles

`task_profile` is a string passed to `POST /monitors/evaluate`. rb-api looks up the profile's weight table and applies it to that evaluation. The SDK does not ship the per-profile weights ŌĆö they live in rb-api.

```python
from reasonblocks import ReasonBlocks

rb = ReasonBlocks(
    api_key="rb_live_...",
    task_profile="coding",   # or "pr_review", "qa"
)
```

Three profiles are defined in `rb-api/app/scoring/monitors.py` `WEIGHTS_BY_PROFILE`:

### `coding` (default)

Standard weights for an agent that edits code, runs tests, and may loop. Identical to the SDK's local `DEFAULT_WEIGHTS`.

| Monitor | Weight |
|---|---|
| `streak` | `0.35` |
| `call_count` | `0.15` |
| `edit_revert` | `0.15` |
| `test_repeat` | `0.15` |
| `diversity` | `0.10` |
| `hedge` | `0.10` |

### `pr_review`

Read-only review agent. `edit_revert` and `test_repeat` are inapplicable (review agents legitimately read many files without writing or running tests), so their weight is redistributed onto `streak` and `diversity` ŌĆö where review agents most often fail by re-explaining the same diff or sprawling across files.

| Monitor | Weight |
|---|---|
| `streak` | `0.45` |
| `call_count` | `0.15` |
| `edit_revert` | `0.0` |
| `test_repeat` | `0.0` |
| `diversity` | `0.20` |
| `hedge` | `0.20` |

### `qa`

QA / tutoring agent. `hedge` dominates (over-qualified answers are the primary failure mode); `call_count` matters less because QA agents make fewer tool calls overall.

| Monitor | Weight |
|---|---|
| `streak` | `0.30` |
| `call_count` | `0.05` |
| `edit_revert` | `0.05` |
| `test_repeat` | `0.10` |
| `diversity` | `0.15` |
| `hedge` | `0.35` |

Resolution order on the server: defaults ŌåÉ named profile preset ŌåÉ per-call `monitor_weights` overrides. Unknown monitor names in overrides are dropped; negative weights are clamped to `0`.

## Per-call monitor weights override

Pass `monitor_weights` on `ReasonBlocksConfig` to override individual monitor weights on top of the chosen profile. The override is forwarded into `MonitorSteeringInjection` and rides on every `POST /monitors/evaluate` request.

```python
from reasonblocks import ReasonBlocksConfig

config = ReasonBlocksConfig(
    monitor_task_profile="coding",
    monitor_weights={
        "streak": 0.5,    # raise ŌĆö earlier loop detection
        "hedge": 0.05,    # lower ŌĆö your agent legitimately hedges
    },
)
```

Partial dicts are fine ŌĆö unspecified monitors fall through to the profile, then to server defaults. Unknown monitor names are dropped server-side; negative weights are clamped to `0`.

<Note>
  `monitor_weights` is only a field on `ReasonBlocksConfig`. The simpler `ReasonBlocks(...)` constructor doesn't expose it ŌĆö use the config + `build_middleware` path when you need per-call weight overrides.
</Note>
