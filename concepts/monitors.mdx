ReasonBlocks ships **two** monitor surfaces. They share the same six heuristics by name but live in different processes and do different jobs. Conflating them is the easiest way to misread what the SDK is doing.

- **Server-side** — `POST /monitors/evaluate` on rb-api. Driven by `MonitorSteeringInjection`. Returns a steering intervention the middleware appends to the system message. **This is what produces the `[REASONBLOCKS]` block users see.**
- **Local** — `reasonblocks.monitors.suite.evaluate_all` and the per-monitor `score_*` functions. Pure Python, no network. Used by `TokenSavingMiddleware` for early-exit decisions and by `SQLiteMonitorSink` for offline trace inspection.

## Server-side: monitor steering

`MonitorSteeringInjection.retrieve` posts the current trace to `POST /monitors/evaluate` with the model identifier, `task_profile`, and any explicit weight overrides. The server runs the trajectory monitor suite, applies its two-path injection gate, optionally falls back to an `E2` pattern lookup, and returns:

```python
{
    "inject":              bool,           # whether to fire an injection this step
    "intervention_text":   str | None,     # rendered text to append to the system message
    "fired":               list[str],      # monitor names whose individual score >= fire threshold
    "scores":              dict[str, float],
    "composite":           float,          # weighted sum
    "failure_type":        str | None,     # categorical failure label
    "intervention_source": str | None,     # which path produced intervention_text
}
```

When `inject` is `True` and the text is new (different from the last firing's signature), the middleware queues a `PendingInjection` carrying that text plus `monitors_fired` and `failure_type`. Per-run guardrails are enforced client-side after the server's decision:

- **Hard cap of 5 monitor injections per run.**
- **Cooldowns by FSM state:** every 2 steps in `SLOW`/`SKIP`, every 3 in `NORMAL`, every 5 in `FAST`.
- **Duplicate suppression:** the same intervention text is not re-queued back-to-back.

The `task_profile` constructor argument selects the server-side weight preset:

| Profile | Notes |
|---|---|
| `coding` (default) | Standard weights matching the local suite. |
| `pr_review` | Read-only review agents — `edit_revert` and `test_repeat` weights drop to 0; `streak`, `diversity`, `hedge` are redistributed. |
| `qa` | QA / tutoring agents — `hedge` dominates; `call_count` and `edit_revert` drop. |

## Local: the six-monitor suite

`reasonblocks.monitors.suite` exposes `evaluate_all(steps)` plus per-monitor `score_*` functions. Each returns a float in `[0, 1]` where 1 means maximum badness. The default fire threshold is **0.6**. The aggregator's default weights are:

```python
DEFAULT_WEIGHTS = {
    "streak":      0.35,
    "call_count":  0.15,
    "edit_revert": 0.15,
    "test_repeat": 0.15,
    "diversity":   0.10,
    "hedge":       0.10,
}
```

The local suite is consumed by:

- `TokenSavingMiddleware`, which reads `evaluate_all`'s `composite` and `fired` to decide when to inject an early-exit nudge or compress stale tool outputs.
- `SQLiteMonitorSink`, which writes per-step monitor scores to a local SQLite DB for offline inspection.

It does **not** drive the steering injection in the system message. That path is server-side only.

## The six monitors

<AccordionGroup>
  <Accordion title="streak — same action repeated consecutively (weight: 0.35)">
    Walks the trace in reverse and counts the current run of identical actions, ignoring steps that produced no action.

    **Score:** `min(run_length / 5, 1.0)`. A streak of 1 scores 0.0; a streak of 5 or more scores 1.0.

    **Fires at:** score `>= 0.6`, i.e. 3 or more consecutive identical actions.

    **Why it carries the highest weight:** action loops are the most common and most costly failure pattern in long-running agents.
  </Accordion>

  <Accordion title="call_count — tool-call budget consumed (weight: 0.15)">
    Counts every step that produced an action.

    **Score:** `min(total_tool_calls / 20, 1.0)`. The score rises linearly from 0.0 at zero calls to 1.0 at 20 calls.

    **Fires at:** score `>= 0.6`, i.e. 12 or more tool calls.

    A high call count alone is not a failure. In combination with another monitor, it surfaces in the composite as "the agent has spent a lot of its budget without finishing."
  </Accordion>

  <Accordion title="edit_revert — edits that undo each other (weight: 0.15)">
    Detects two patterns of file-editing thrash:

    1. **Content revert:** the most recent edit's content is closer (by character-trigram Jaccard) to the edit *before* the previous one than to the previous edit itself, by a margin of 0.10.
    2. **Edit-fail-edit cycles:** the agent edits a file, receives an error observation, then edits the same path again — repeated two or more times.

    **Score:** `1.0` on a clean revert or two cycles on the same path; `0.0` on a clean edit history.

    Recognized edit-like tools: `edit`, `write`, `str_replace`, `str_replace_editor`, `patch`, `apply_patch`, `create_file`, `overwrite`.
  </Accordion>

  <Accordion title="test_repeat — same test failure repeating (weight: 0.15)">
    Detects when the agent runs a test, receives a failure, and then runs the same test again with the same normalized failure signature.

    Volatile fragments are stripped before comparison: temp paths, UUIDs, hex memory addresses, PIDs, timestamps, and large numeric runs.

    **Score:** `1.0` when the same normalized signature appears two or more times consecutively; `0.0` otherwise.

    Test steps are identified by tool name (`pytest`, `test`, `run_tests`, `npm test`, `cargo test`) or by error keywords in the observation (`assertionerror`, `failed`, `failures=`).
  </Accordion>

  <Accordion title="diversity — collapsed tool exploration (weight: 0.10)">
    Detects when recent tool usage has narrowed to two or fewer distinct tools across a 5-call tail, after the run has accumulated at least 8 total tool calls.

    **Score:**
    - `1.0` if only one tool has been used in the last 5 calls.
    - `0.7` if exactly two tools but the pattern is not a healthy alternation (`a-b-a-b-a` is excluded).
    - `0.0` otherwise, including before the 8-call minimum is reached.

    **Fires at:** score `>= 0.6`, covering both the single-tool (1.0) and non-alternating two-tool (0.7) cases.
  </Accordion>

  <Accordion title="hedge — rising hedging or explicit retraction (weight: 0.10)">
    Operates on the `thought` text of each step.

    1. **Retraction phrases** ("I was wrong", "never mind", "disregard that", "the bug is actually not...") anywhere in the trace score `1.0` immediately.
    2. **Hedge density ratio:** the density of hedging words ("maybe", "perhaps", "might", "not sure", "on second thought", ...) in the late half of the run vs the early half. A ratio at or below 2× scores 0.0; ratio of 4× scores 1.0; linear in between.

    With fewer than 4 thoughts there is not enough history for a ratio, so the score falls back to raw hedge density.
  </Accordion>
</AccordionGroup>

## Composite score and the local fire threshold

`evaluate_all` returns the per-monitor `scores`, the weighted `total` (the composite), the list of `fired` monitors (those with score `>= 0.6`), and the resolved `weights`.

```
composite = 0.35 * streak
          + 0.15 * call_count
          + 0.15 * edit_revert
          + 0.15 * test_repeat
          + 0.10 * diversity
          + 0.10 * hedge
```

You can override weights or the fire threshold per call:

```python
from reasonblocks import evaluate_all, DEFAULT_WEIGHTS

result = evaluate_all(
    steps,
    weights={**DEFAULT_WEIGHTS, "streak": 0.45, "diversity": 0.05},
    fire_threshold=0.5,
)
```

## How monitor results gate E1 retrieval

The middleware consumes the **server-side** evaluation (not the local suite) to gate E1 retrieval. The two-step lookback keeps the gate open across consecutive at-risk steps.

```
allow E1 if:
  any monitor fired this step
  OR composite this step > 0.15
  OR any monitor fired on the previous 2 steps
```

If `MonitorSteeringInjection` is not configured at all, E1 is allowed unconditionally for backward compatibility.

<Tip>
  Monitor injection text is also subject to the per-run cap (5) and FSM-state cooldowns (every 2 steps in `SLOW`/`SKIP`, every 3 in `NORMAL`, every 5 in `FAST`). These guardrails are independent of the E1 gate.
</Tip>
